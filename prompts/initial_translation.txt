6. Frama-C Compatibility Rules (Mandatory)
   - Output Frama-C-compatible ACSL only, not generic pseudo-ACSL.
   - Use only `/*@ ... */` and `//@ ...` annotation forms.
   - Never put ordinary `// ...` comments on the same line as a `//@` annotation.
   - Never use chained inequalities such as `0 <= i <= n`; always write `0 <= i && i <= n`.
   - Do not use unsupported constructs such as `\length(pointer)` on raw C pointers.
   - Do not invent logic helpers unless you define them explicitly in valid ACSL.
   - Keep one ACSL assertion per annotation comment.
   - Prefer simple Frama-C-friendly annotations over compact but ambiguous formulas.

7. ACSL Syntax Guide (Use These Patterns)
   - Function contract: `/*@ requires P; assigns A; ensures Q; */`
   - Loop contract: `/*@ loop invariant I; loop assigns A; loop variant V; */`
   - Assertion: `//@ assert P;`
   - Quantifier: `\forall integer i; range ==> property`
   - Pointer validity: `\valid(p)` or `n == 0 || \valid_read(p + (0 .. n-1))`
   - Return-value reference: `\result`
   - Pre-state reference: `\old(expr)`

8. Invalid Patterns to Avoid
   - `\length(pointer)` or `\strlen(pointer)`
   - `0 <= i <= n`
   - undeclared helper predicates/functions used from contracts
   - multiple ACSL assertions in one annotation comment
   - prose or markdown fences inside the code output

9. Output Requirements
    - Output ONLY the final C code.
    - Do NOT include explanations.
    - Do NOT include a main function.
   - Include necessary headers (e.g., <limits.h>, <stddef.h>) only if required.
   - Ensure the result is syntactically correct C code and suitable for formal verification.
